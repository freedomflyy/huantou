# 幻头项目后端系统说明报告

最后更新：`2026-03-13`

## 1. 报告目的

这份报告用于说明当前“幻头”项目后端系统已经具备的能力、系统结构、业务流程、接口分组、核心状态，以及这些能力会如何影响前端页面设计与交互设计。

阅读目标不是看代码细节，而是回答下面几个问题：

- 当前后端到底能做什么，哪些功能是真实可用的。
- 当前后端的数据流和任务流长什么样。
- 前端页面应该围绕哪些对象来设计。
- 哪些前端状态是必须有的，哪些是可选增强。
- 哪些能力已经可以按“正式产品逻辑”去设计，而不是测试原型。

---

## 2. 后端系统定位

当前后端是一个围绕“AI 头像生成与编辑”构建的业务后端，技术栈为：

- `FastAPI`
- `PostgreSQL`
- `JWT access_token + refresh_token`
- `COS / 本地存储`
- `火山引擎图片生成 API`
- `腾讯 COS/CI 审核（可切换）`

从产品角度看，它不是一个单纯的“文生图接口代理”，而是一个完整的业务系统，已经覆盖：

- 用户登录
- 用户积分体系
- 图像生成/编辑任务体系
- 资产入库与过期管理
- 收藏能力
- 审核拦截与审核回调
- 管理后台接口
- 日志、限流、指标、错误追踪的基础设施

也就是说，前端设计现在不应该只围绕“生成页面”做，而应该围绕完整产品闭环设计。

---

## 3. 当前后端已实现的产品能力

### 3.1 用户与登录

已实现：

- 微信登录：`POST /api/v1/auth/wechat-login`
- JWT 登录态：`access_token + refresh_token`
- 刷新登录态：`POST /api/v1/auth/refresh`
- 登出当前会话：`POST /api/v1/auth/logout`
- 登出全部会话：`POST /api/v1/auth/logout-all`

业务含义：

- 用户第一次登录会创建账号。
- 新用户自动获得注册积分。
- 每日首次登录自动获得每日积分。
- 后端已经把“账号体系 + 登录奖励 + 会话管理”统一到一个流程里。

对前端的含义：

- 前端必须有“登录前 / 登录后”的状态分层。
- 登录成功后，首页可以直接展示积分、昵称、头像等用户信息。
- 前端要准备好“token 失效重新登录”的体验，而不是只做一次登录。

### 3.2 积分系统

已实现：

- 当前积分余额：`GET /api/v1/points/balance`
- 积分流水：`GET /api/v1/points/ledgers`
- 创建任务时自动扣积分
- 任务失败时自动返还积分
- 管理员手动调整积分

当前积分规则：

- 新用户赠送积分
- 每日登录赠送积分
- `txt2img`、`img2img`、`style_transfer` 有固定消耗
- `quick_edit` 走任务体系，但本质是本地图片编辑流程

业务含义：

- 积分不是装饰字段，而是整个产品的核心资源计费对象。
- 每一个生成类入口都应该显式展示消耗。
- “积分不足”是前端必须处理的业务错误之一。

对前端的含义：

- 首页、创作页、确认提交页、结果页都应该有积分感知。
- 前端最好统一做一个“积分展示模块”，放在首页头部、个人中心、提交前确认区域都能复用。
- 需要设计积分不足提示、积分流水页、管理员调积分后积分变化的展示逻辑。

### 3.3 任务系统

当前任务类型共有 4 类：

- `txt2img`：文生图
- `img2img`：单图修改
- `style_transfer`：双图参考生成
- `quick_edit`：基础快速编辑

已实现接口：

- 创建任务：`POST /api/v1/tasks`
- 查询任务列表：`GET /api/v1/tasks`
- 查询任务详情：`GET /api/v1/tasks/{task_id}`
- 执行任务：`POST /api/v1/tasks/{task_id}/execute`
- 重试失败任务：`POST /api/v1/tasks/{task_id}/retry`

任务状态：

- `pending`
- `queued`
- `running`
- `succeeded`
- `failed`
- `canceled`

业务含义：

- 当前系统的核心设计是“先创建任务，再执行任务”，而不是前端一键发请求后直接拿图。
- 所有正式创作能力都可以统一映射到任务中心和历史列表。
- 任务不是一次性请求，而是产品级对象，包含重试、失败、返还、来源、结果等完整信息。

对前端的含义：

- 前端必须把“任务”当成一级对象设计。
- 至少应该有：任务创建态、执行中态、成功态、失败态。
- 历史任务页不是附属页面，而是产品核心页面之一。
- 结果页不应该只展示图片，还应该能表达任务类型、参数摘要、失败原因、重试入口。

## 4. 四类核心创作能力说明

### 4.1 文生图 `txt2img`

输入：

- `prompt`
- 可选 `negative_prompt`
- 可选参数 `params`

输出：

- 任务成功
- 一张或多张输出图片 URL
- 同时落资产记录

适合前端页面：

- 纯文本输入页
- 提示词模板页
- 提交确认弹层
- 结果页

### 4.2 图片修改 `img2img`

输入：

- `input_image_url`
- `prompt`
- 可选 `negative_prompt`
- 可选参数 `params`

输出：

- 新结果图
- 新资产记录

适合前端页面：

- 上传图 + 文本说明页面
- 上传成功后预览页面
- 参数调节页面

### 4.3 参考图生成 `style_transfer`

输入：

- `input_image_url`
- `reference_image_url`
- `prompt`
- 可选参数 `params`

输出：

- 基于主图与参考图生成的结果图

适合前端页面：

- 双图上传布局
- 主图 / 参考图明确分区
- 生成规则说明区

这是非常重要的一点：  
前端在设计这个页面时，必须把“双图输入”作为主结构，而不是单图页的变种。

### 4.4 快速编辑 `quick_edit`

当前不是调用外部 AI 生成，而是本地图片处理流水线，已支持：

- 裁剪
- 缩放
- 旋转
- 水平翻转 / 垂直翻转
- 饱和度
- 亮度
- 对比度
- 锐化
- 模糊
- 输出格式与质量控制

业务含义：

- 这是目前最接近“即时编辑器”的能力。
- 它和文生图任务不同，更适合做成“编辑器体验”。

对前端的含义：

- 快速编辑页应明显区别于 AI 生成页。
- 这里适合做滑杆、预设、裁切框、旋转控件、导出确认。
- 这条能力未来也可以自然扩展成完整编辑器。

## 5. 资产系统说明

资产接口：

- 上传图片：`POST /api/v1/assets/upload`
- 我的作品列表：`GET /api/v1/assets`
- 收藏列表：`GET /api/v1/assets/favorites`
- 收藏作品：`POST /api/v1/assets/{asset_id}/favorite`
- 取消收藏：`DELETE /api/v1/assets/{asset_id}/favorite`
- 本地图片访问：`GET /api/v1/assets/local/{object_key}`

资产的产品定义：

- 用户上传的原图
- 任务生成的结果图
- 快速编辑后的图片
- 资产有来源任务、存储位置、收藏态、是否下架、是否过期等信息

当前规则：

- 结果图默认保留 `7` 天
- 成功任务会自动写入资产表
- 收藏是独立关系表，不是简单布尔值
- 管理员可以下架资产

对前端的含义：

- “作品”与“任务结果”可以统一，但最好在展示层区分来源。
- 资产页应该支持：缩略图网格、详情预览、收藏、取消收藏、过期说明。
- 收藏页、作品页、首页灵感区都可以共用一套作品卡片组件。

---

## 6. 审核系统说明

当前后端已经接入审核体系，支持：

- 文本审核
- 输入图审核
- 输出图审核
- 腾讯审核回调接收
- 审核记录入库
- 命中违规后自动下架作品

审核位置：

- 创建任务前，审核 `prompt / negative_prompt`
- 执行任务前，审核输入图
- 生成结果后，审核输出图
- 上传资产时，审核上传图片

业务含义：

- 前端不能假设“提交即成功”。
- 用户会遇到“提示词不通过”“图片不通过”“结果被拦截”的情况。

对前端的含义：

- 必须为审核失败准备清晰提示，不要只显示“请求失败”。
- 需要在上传页、生成页、结果页中预留审核相关提示文案。
- 后续如果加入用户申诉或人工复核入口，也有后端延展空间。

---

## 7. 管理后台能力说明

当前管理端接口已经具备：

- 总览统计：`GET /api/v1/admin/overview`
- 用户列表：`GET /api/v1/admin/users`
- 用户状态变更：`POST /api/v1/admin/users/{user_id}/status`
- 积分调整：`POST /api/v1/admin/users/{user_id}/points-adjust`
- 管理任务列表：`GET /api/v1/admin/tasks`
- 管理任务重试：`POST /api/v1/admin/tasks/{task_id}/retry`
- 管理作品列表：`GET /api/v1/admin/assets`
- 作品下架：`POST /api/v1/admin/assets/{asset_id}/take-down`
- 审核审计列表：`GET /api/v1/admin/moderation-audits`
- 简易 Web 控制台：`GET /api/v1/admin/console`

业务含义：

- 后端不是只做用户端接口，管理端已经具备最小闭环。
- 管理后台后续可以独立做 Web，不建议塞进小程序。

对前端的含义：

- 用户端 UI 可以先不展示这些能力。
- 但在作品详情、失败任务、账号异常等页面设计时，要知道后端已经支持人工介入与运营处理。

## 8. 系统架构与模块划分

后端当前可以粗分为 7 个层次：

### 8.1 API 路由层

负责：

- 参数接收
- 鉴权
- 业务入口
- 响应输出

主要模块：

- `auth`
- `points`
- `tasks`
- `assets`
- `admin`
- `moderation`
- `health`

### 8.2 服务层

负责：

- 生成任务执行
- 快速编辑
- 火山引擎调用
- 图片下载与入库
- 审核
- 存储
- token 签发与刷新

这层基本就是产品规则的核心。

### 8.3 数据模型层

核心对象：

- `User`
- `GenerationTask`
- `Asset`
- `AssetFavorite`
- `PointsLedger`
- `RefreshToken`
- `ModerationAudit`

这几个对象基本就是前端信息架构的基础。

### 8.4 存储层

支持：

- 本地存储
- COS 存储

当前建议用 COS，适合正式使用。

### 8.5 审核层

支持：

- `mock`
- `tencent`

前期联调可以 `mock`，正式联调/上线使用腾讯。

### 8.6 运行与安全层

包含：

- 结构化日志
- `X-Request-Id`
- 限流
- 全局异常处理
- Sentry

### 8.7 运维脚本层

已提供脚本用于：

- 初始化数据库
- 跑全链路 demo
- 验证云服务
- 清理过期资源
- 清理过期 refresh token
- 执行任务 worker

## 9. 当前真实可跑的后端业务闭环

从用户第一次打开产品，到最终拿到作品，当前真实流程是：

1. 用户微信登录
2. 后端创建或更新用户
3. 发放注册积分或每日积分
4. 前端读取积分余额
5. 用户选择任务类型
6. 用户上传图片或填写提示词
7. 前端创建任务
8. 后端扣除积分
9. 前端发起执行
10. 后端审核文本和输入图
11. 后端调用火山引擎或快速编辑服务
12. 后端审核输出图
13. 后端把结果图入存储并写入资产表
14. 前端读取任务结果和作品列表
15. 用户收藏、查看、重试或继续编辑

这条链路已经不是原型级逻辑，而是产品级逻辑。

## 10. 前端设计时必须考虑的状态

这一部分很关键。  
如果前端设计脱离这些状态，页面就会“好看但不好用”。

### 10.1 登录态

- 未登录
- 登录中
- 登录成功
- token 失效重新登录

### 10.2 积分态

- 积分正常
- 积分不足
- 赠分到账
- 扣分成功
- 失败返还

### 10.3 任务态

- 未创建
- 已创建未执行
- 排队中
- 执行中
- 成功
- 失败
- 可重试

### 10.4 资源态

- 无输入图
- 上传中
- 上传成功
- 审核失败
- 预览失败
- 结果图可浏览
- 结果图已过期

### 10.5 审核态

- 提示词通过
- 提示词拦截
- 输入图通过
- 输入图拦截
- 输出图通过
- 输出图拦截

## 11. 对前端页面设计的直接建议

基于当前后端，前端页面应该至少划分为以下模块：

### 11.1 首页

建议承载：

- 用户欢迎信息
- 当前积分
- 四大功能入口
- 最近作品 / 灵感位
- 最近任务状态摘要

### 11.2 创作入口层

建议做成统一创作中心，然后分流到：

- 文生图
- 图片修改
- 参考图生成
- 快速编辑

### 11.3 任务执行层

建议单独设计：

- 提交确认态
- 执行中 loading
- 失败态
- 重试态

### 11.4 结果层

建议包含：

- 出图展示
- 来源任务信息
- 收藏按钮
- 再次创作
- 继续编辑
- 返回资产库

### 11.5 资产层

建议拆分为：

- 我的作品
- 收藏作品
- 上传素材
- 作品详情

### 11.6 个人中心

建议承载：

- 用户信息
- 当前积分
- 积分流水入口
- 我的任务入口
- 我的作品入口
- 服务说明入口

### 11.7 任务历史页

建议突出：

- 任务类型
- 状态
- 创建时间
- 是否失败
- 是否可重试

## 12. 当前后端对前端最友好的地方

有几项能力会明显降低前端复杂度：

- 任务模型已经统一，四类能力可以共用一套任务中心逻辑。
- 资产模型已经统一，上传图、结果图、编辑图都可以用统一作品卡设计。
- 积分规则后端已经给出，前端不需要自己维护业务规则。
- 失败返还逻辑后端已经做了，前端不用自己判断是否返分。
- 管理后台能力已经独立，不必塞进用户端小程序。

---

## 13. 当前后端对前端最需要注意的地方

前端一定要注意这几个现实约束：

- 任务不是即时成功，要设计排队和执行中的体验。
- 审核会打断流程，不能只做理想路径。
- COS 图片可能因为权限或过期导致加载失败，前端要有占位图回退。
- 作品默认 7 天过期，因此作品页和详情页最好有“有效期”表达。
- 当前 AI 结果数不保证固定张数，结果页不能写死“4 张图”布局。

---

## 14. 当前系统完成度判断

如果以“是否能支撑正式前端设计”为标准，当前后端已经足够：

- 能支撑首页设计
- 能支撑创作流设计
- 能支撑任务系统设计
- 能支撑作品系统设计
- 能支撑个人中心设计
- 能支撑后续后台管理 Web 设计

换句话说，现在前端完全可以按照“正式产品信息架构”来做，而不是测试页拼接。

---

## 15. 建议的前端设计优先级

如果前端要基于后端真实能力来设计，我建议优先级如下：

1. 首页
2. 创作中心
3. 四类任务页
4. 执行中页
5. 结果页
6. 作品/收藏页
7. 任务历史页
8. 个人中心
9. 积分流水页

理由很简单：

- 这条顺序与后端核心闭环一致。
- 先把“登录 -> 创作 -> 出图 -> 沉淀作品”跑顺，产品就活了。

---

## 16. 一句话结论

当前后端已经不是“接口雏形”，而是一个可以支撑正式产品设计的业务底座。  
前端现在最应该做的，不是继续验证后端能不能跑，而是围绕“任务 + 资产 + 积分”这三条主线，设计一套真正统一的产品界面。
